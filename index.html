Capa 1 de hoy

La HU de Validación

Antes de guardar un dato hay que asegurarse de que es válido: campos obligatorios, con el formato correcto y del tamaño permitido. Esta es la historia de usuario del docente.

USR-V01 Crear la validación de Usuario: interfaz IUsuarioValidador e implementación Validación
Criterios de aceptación
  • La interfaz IUsuarioValidador declara los métodos de validación de Usuario.
  • Obligatoriedad — validarNombreObligatorio(String): el nombre no puede ser null ni vacío/en blanco.
  • Formato — validarCorreoFormato(String): el correo debe cumplir un patrón de email válido.
  • Tamaño — validarContrasenaHashLongitud(String): el hash BCrypt debe tener 60 caracteres.
  • Método validar(Usuario) que ejecuta las 3 reglas.
  • La clase UsuarioValidadorImpl implementa la interfaz, va anotada con @Component y lanza ResponseStatusException(HttpStatus.BAD_REQUEST, ...) con un mensaje claro (Spring responde 400).

¿Por qué una capa aparte y no validar “a lo bruto” en el controlador?

  • Responsabilidad única: el validador solo sabe de reglas; el servicio, de negocio.
  • Reutilizable: la misma validación sirve al crear y al actualizar.
  • Testeable: puedes probar cada regla por separado, sin base de datos ni HTTP.
USR-V01
El contrato

La interfaz IUsuarioValidador

La interfaz es el contrato: dice qué se valida, no cómo. Empieza con I por convención de clase. Para señalar datos inválidos usamos la excepción que ya trae Spring Web: ResponseStatusException.

IUsuarioValidador.java · el contrato 
public interface IUsuarioValidador {

    // Una regla por método: claro y testeable.
    void validarNombreObligatorio(String nombre);
    void validarCorreoFormato(String correo);
    void validarContrasenaHashLongitud(String contrasenaHash);

    // Orquesta las 3 reglas sobre la entidad completa.
    void validar(Usuario usuario);
}
ResponseStatusException · sin clases propias 
// Spring Web YA trae la excepción: no creas ninguna clase.
// 1er argumento = STATUS HTTP · 2do argumento = mensaje.
import org.springframework.http.HttpStatus;
import org.springframework.web.server.ResponseStatusException;

// Dato inválido -> 400 Bad Request (se usa dentro del Impl):
throw new ResponseStatusException(
        HttpStatus.BAD_REQUEST, "El nombre es obligatorio.");

Interfaz = contrato

Declara métodos sin cuerpo. Otros pueden depender de IUsuarioValidador sin conocer la implementación.

Una regla por método

Métodos pequeños y con nombre claro. validar(Usuario) solo las encadena.

ResponseStatusException

La trae Spring Web. Lanzas el status HTTP directo (aquí 400) con su mensaje. Sin clases extra ni @ExceptionHandler.

USR-V01
El cómo

La clase UsuarioValidadorImpl

Aquí sí está la lógica. La clase implements IUsuarioValidador, va anotada con @Component (para que Spring la inyecte) y cada regla lanza ResponseStatusException con HttpStatus.BAD_REQUEST (400) si el dato no cumple.

UsuarioValidadorImpl.java 
import java.util.regex.Pattern;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Component;
import org.springframework.web.server.ResponseStatusException;

@Component   // Spring lo registra como bean → se puede inyectar
public class UsuarioValidadorImpl implements IUsuarioValidador {

    private static final Pattern CORREO =
        Pattern.compile("^[\\w.+-]+@[\\w-]+\\.[\\w.-]+$");

    @Override
    public void validarNombreObligatorio(String nombre) {
        if (nombre == null || nombre.isBlank()) {
            throw new ResponseStatusException(
                HttpStatus.BAD_REQUEST, "El nombre es obligatorio.");
        }
    }

    @Override
    public void validarCorreoFormato(String correo) {
        if (correo == null || !CORREO.matcher(correo).matches()) {
            throw new ResponseStatusException(
                HttpStatus.BAD_REQUEST, "El correo no tiene un formato valido.");
        }
    }

    @Override
    public void validarContrasenaHashLongitud(String contrasenaHash) {
        if (contrasenaHash == null || contrasenaHash.length() != 60) {
            throw new ResponseStatusException(HttpStatus.BAD_REQUEST,
                "La contrasena_hash debe ser un hash BCrypt de 60 caracteres.");
        }
    }

    @Override
    public void validar(Usuario usuario) {          // encadena las 3 reglas
        validarNombreObligatorio(usuario.getNombre());
        validarCorreoFormato(usuario.getCorreo());
        validarContrasenaHashLongitud(usuario.getContrasena_hash());
    }
}

@Component

Convierte la clase en un bean de Spring. Así el servicio la puede recibir por constructor sin crearla con new.

throw = corta el flujo

En cuanto una regla falla, validar() lanza y no sigue. El servicio no llegará a guardar un dato inválido.

Ojo con el orden

validar(Usuario) corre las reglas en orden y se detiene en la primera que falle. Por eso el mensaje que ves es el de la primera regla incumplida.

Pruébalo

El validador, en vivo

Escribe valores (o usa un ejemplo) y pulsa Validar. Se ejecutan las mismas 3 reglas del UsuarioValidadorImpl: verás cuáles pasan y, si algo falla, el ResponseStatusException con su 400.

validador.validar(usuario)

Formato · correo con patrón válidoesperando…

La contraseña llega en texto plano al servicio; este la cifra con BCrypt (un hash de 60 caracteres) y esa es la que valida la regla de tamaño.

Capa 2 de hoy

La HU de DTO

El DTO (Data Transfer Object) es lo que la API recibe y devuelve, en vez de la entidad JPA. Aquí los hacemos a mano con POO, sin librerías de mapeo.

USR-D01 Crear los DTO de Usuario (Request y Response) con conversión POO, sin mapper DTO
Criterios de aceptación
  • UsuarioRequestDTO: lo que el cliente envía (nombre, correo, contraseña en texto plano, rol).
  • UsuarioResponseDTO: lo que la API devuelve (id, nombre, correo, rol, activo, colorAvatar, fechaRegistro).
  • El Response NUNCA incluye contrasena_hash ni la contraseña.
  • Son POJOs planos (campos privados + getters/setters, o un record), sin anotaciones JPA.
  • Conversión POO: toEntity() en el Request y static fromEntity(...) en el Response.
  • Sin MapStruct ni ModelMapper: el mapeo es código Java explícito.

Regla de oro del DTO

Nunca devuelvas la entidad por la API. Te evita la recursión infinita (Tutorial 1), no filtra campos sensibles como contrasena_hash y desacopla tu API de la base de datos.

USR-D01
La razón de fondo

¿Por qué usar DTO?

Podrías devolver la entidad Usuario directamente… y te meterías en problemas. El DTO existe para separar lo que guardas (entidad) de lo que expones (API). Estas son las razones:

1. Seguridad

No filtras campos sensibles. El contrasena_hash jamás sale porque el ResponseDTO ni lo tiene.

2. Sin recursión infinita

Las entidades bidireccionales (Usuario↔Prenda) rompen la serialización JSON (Tutorial 1). El DTO plano corta el ciclo.

3. Desacople

Tu API no queda atada a la estructura de la BD. Cambias una columna sin romperle el contrato al cliente.

4. Control fino

Decides qué entra (Request) y qué sale (Response). El cliente no puede mandar id ni activo.

5. Contrato estable

Puedes versionar la API o exponer nombres distintos (colorAvatar) sin tocar la entidad.

6. Payload justo

Envías solo lo necesario: respuestas más pequeñas y rápidas, sin relaciones gigantes que nadie pidió.

Sin DTO — devolver la entidad

  • Expone contrasena_hash y campos internos.
  • Recursión infinita → 500.
  • El cliente ve/manda cualquier campo de la tabla.
  • Cambiar la BD rompe el JSON del cliente.

Con DTO — objeto de transporte

  • Solo expones lo que decides.
  • Plano y finito: sin ciclos.
  • Request y Response separados y claros.
  • La entidad evoluciona sin afectar la API.
USR-D01
Dos objetos, dos usos

Request vs Response

Usamos record (POJO plano inmutable de Java) para no escribir getters a mano. Fíjate en la diferencia clave: el Request trae contraseña; el Response jamás.

UsuarioRequestDTO.java · lo que ENTRA 
// Lo que el cliente ENVÍA al registrar/actualizar.
public record UsuarioRequestDTO(
    String nombre,
    String correo,
    String contrasena,   // texto plano; el servicio la cifra
    String rol
) {}

Sin id (lo genera la BD) y sin activo/fecha_registro (los pone el servidor).

UsuarioResponseDTO.java · lo que SALE 
// Lo que la API DEVUELVE. Sin contrasena_hash.
public record UsuarioResponseDTO(
    UUID id,
    String nombre,
    String correo,
    String rol,
    Boolean activo,
    String colorAvatar,
    LocalDateTime fechaRegistro
) {}

Ni contrasena_hash ni la contraseña. Solo lo que el cliente puede ver.

CampoEntidad UsuarioRequestDTOResponseDTO
idUUID (PK)—✓
nombre / correo / rol✓✓✓
contrasena (plano)—✓ (entra)—
contrasena_hash✓ (cifrada)—✕ nunca
activo / colorAvatar / fechaRegistro✓—✓

¿Y si la entidad tiene relaciones?

En el DTO, las relaciones se representan por el id (UUID) de la entidad asociada, no como objetos anidados. Ej.: una Prenda llevaría idVendedor (UUID), no un Usuario completo. Así el DTO queda plano y sin ciclos.

USR-D01
La duda del código

record vs class

Un record (Java 16+) es una clase inmutable pensada para llevar datos. En una línea te da lo que en una clase son ~40. Por eso es ideal para los DTO.

Como record · 3 líneas 
public record UsuarioResponseDTO(
    UUID id, String nombre, String correo, String rol
) {}

// El compilador GENERA por ti:
//   · constructor con los 4 campos
//   · accesores: id(), nombre(), correo(), rol()
//   · equals(), hashCode() y toString()
//   · campos private final (inmutables)
Como class · lo mismo, a mano 
public class UsuarioResponseDTO {
    private final UUID id;
    private final String nombre;
    private final String correo;
    private final String rol;

    public UsuarioResponseDTO(UUID id, String nombre,
                              String correo, String rol) {
        this.id = id;   this.nombre = nombre;
        this.correo = correo;   this.rol = rol;
    }
    public UUID getId()      { return id; }
    public String getNombre(){ return nombre; }
    public String getCorreo(){ return correo; }
    public String getRol()   { return rol; }
    // + equals(), hashCode() y toString() a mano…
}

Usa record para los DTO

Un DTO solo transporta datos y no cambia una vez creado. record = menos código, inmutable y con equals por valor. Perfecto.

La entidad NO puede ser record

JPA necesita un constructor vacío y campos mutables (con setters) para hidratar la entidad desde la BD. Por eso Usuario es una clase y el DTO es un record.

USR-D01
Sin mapper

Convertir a mano, con POO

“Sin mapper” significa: nada de MapStruct ni ModelMapper. La conversión entidad↔DTO la escribes tú, con métodos normales. Dos métodos y listo: toEntity() y fromEntity(...).

UsuarioRequestDTO · toEntity() 
public record UsuarioRequestDTO(
    String nombre, String correo,
    String contrasena, String rol
) {
    // DTO -> entidad. OJO: la contrasena NO se copia aquí;
    // el servicio la cifra y setea el hash aparte.
    public Usuario toEntity() {
        Usuario u = new Usuario();
        u.setNombre(nombre);
        u.setCorreo(correo);
        u.setRol(rol);
        u.setActivo(true);
        return u;
    }
}
UsuarioResponseDTO · fromEntity() 
public record UsuarioResponseDTO(
    UUID id, String nombre, String correo, String rol,
    Boolean activo, String colorAvatar, LocalDateTime fechaRegistro
) {
    // entidad -> DTO. Método estático de fábrica.
    // NUNCA copia contrasena_hash.
    public static UsuarioResponseDTO fromEntity(Usuario u) {
        return new UsuarioResponseDTO(
            u.getId(), u.getNombre(), u.getCorreo(), u.getRol(),
            u.getActivo(), u.getColor_avatar(), u.getFecha_registro()
        );
    }
}

Ventajas de hacerlo a mano

  • Cero dependencias y cero “magia”: ves exactamente qué se copia.
  • Perfecto para aprender POO: constructores, métodos estáticos de fábrica.
  • Controlas qué se oculta (el hash) sin configurar nada.

¿Y el mapper (MapStruct)?

Existe y en proyectos grandes ahorra código repetitivo (genera el mapeo en compilación). Pero aquí no lo usamos: primero hay que entender el mapeo a mano. El mapper es un atajo para después.

Capa 3 de hoy

La HU de Servicio

El servicio es el director de orquesta: recibe DTOs, valida, usa el repositorio y devuelve DTOs. Y sus dependencias entran por el constructor.

USR-S01 Crear el servicio de Usuario: interfaz IUsuarioServicio e implementación con inyección por constructor Servicio
Criterios de aceptación
  • IUsuarioServicio declara: registrar, listar, buscarPorId, actualizar, eliminar (sobre DTOs).
  • UsuarioServicioImpl va anotada con @Service e implementa la interfaz.
  • Inyección por constructor: UsuarioRepository, IUsuarioValidador y PasswordEncoder como private final, sin @Autowired en el campo.
  • registrar valida, comprueba existsByCorreo (si ya existe → 409 Conflict), cifra la contraseña y guarda.
  • Devuelve UsuarioResponseDTO (nunca el hash ni la entidad).
  • buscarPorId/actualizar/eliminar lanzan ResponseStatusException(HttpStatus.NOT_FOUND, ...) (404) si el id no existe.
USR-S01
La clave del servicio

Inyección por constructor

Inyectar = que Spring te entregue las dependencias ya creadas, en vez de hacer new. Hay dos formas; usamos la recomendada: por constructor.

Por CAMPO · desaconsejada 
@Service
public class UsuarioServicioImpl implements IUsuarioServicio {

    @Autowired                       // inyección por CAMPO
    private UsuarioRepository repo;  // no puede ser final

    // ...
}

No es final, cuesta más testear (necesitas Spring/reflection) y esconde cuántas dependencias tiene la clase.

Por CONSTRUCTOR · recomendada 
@Service
public class UsuarioServicioImpl implements IUsuarioServicio {

    private final UsuarioRepository repo;
    private final IUsuarioValidador validador;
    private final PasswordEncoder encoder;

    // Un solo constructor → Spring inyecta solo (sin @Autowired).
    public UsuarioServicioImpl(UsuarioRepository repo,
                               IUsuarioValidador validador,
                               PasswordEncoder encoder) {
        this.repo = repo;
        this.validador = validador;
        this.encoder = encoder;
    }
}

Campos final (inmutables), fácil de testear (le pasas mocks al constructor) y las dependencias quedan a la vista.

Regla práctica

Si la clase tiene un único constructor, Spring inyecta sus parámetros automáticamente — ni siquiera necesitas @Autowired. Campos private final + constructor = el estándar hoy.

USR-S01
El contrato del negocio

La interfaz IUsuarioServicio

Igual que en validación: primero el contrato. Fíjate en que todo entra y sale como DTO, nunca como entidad Usuario.

IUsuarioServicio.java 
public interface IUsuarioServicio {

    UsuarioResponseDTO       registrar(UsuarioRequestDTO dto);
    List<UsuarioResponseDTO> listar();
    UsuarioResponseDTO       buscarPorId(UUID id);
    UsuarioResponseDTO       actualizar(UUID id, UsuarioRequestDTO dto);
    void                     eliminar(UUID id);
}

Entra

Un UsuarioRequestDTO (o un UUID para buscar/borrar).

Sale

Un UsuarioResponseDTO (o una lista). Nunca la entidad.

Solo firmas

La interfaz no tiene lógica. El Impl pone el cómo.

USR-S01
Todo junto

UsuarioServicioImpl

Aquí se juntan las 3 capas: el servicio construye la entidad desde el DTO, la cifra, la valida, la guarda con el repositorio y devuelve un DTO.

UsuarioServicioImpl.java 
import org.springframework.http.HttpStatus;
import org.springframework.web.server.ResponseStatusException;

@Service
public class UsuarioServicioImpl implements IUsuarioServicio {

    private final UsuarioRepository repo;
    private final IUsuarioValidador validador;
    private final PasswordEncoder encoder;

    public UsuarioServicioImpl(UsuarioRepository repo,
                               IUsuarioValidador validador,
                               PasswordEncoder encoder) {
        this.repo = repo;
        this.validador = validador;
        this.encoder = encoder;
    }

    @Override
    public UsuarioResponseDTO registrar(UsuarioRequestDTO dto) {
        // 1) DTO -> entidad (POO, sin mapper)
        Usuario usuario = dto.toEntity();

        // 2) Cifro la contrasena en texto plano -> hash BCrypt (60 chars)
        usuario.setContrasena_hash(encoder.encode(dto.contrasena()));

        // 3) Reglas de negocio + validación
        if (repo.existsByCorreo(usuario.getCorreo())) {
            throw new ResponseStatusException(          // 409 Conflict
                HttpStatus.CONFLICT, "El correo ya esta registrado.");
        }
        validador.validar(usuario);   // obligatoriedad, formato, tamaño (400)

        // 4) Persisto y devuelvo un DTO (nunca la entidad ni el hash)
        Usuario guardado = repo.save(usuario);
        return UsuarioResponseDTO.fromEntity(guardado);
    }

    @Override
    public List<UsuarioResponseDTO> listar() {
        return repo.findAll().stream()
                   .map(UsuarioResponseDTO::fromEntity)
                   .toList();
    }

    @Override
    public UsuarioResponseDTO buscarPorId(UUID id) {
        Usuario usuario = repo.findById(id)
            .orElseThrow(() -> new ResponseStatusException(   // 404 Not Found
                HttpStatus.NOT_FOUND, "Usuario no encontrado: " + id));
        return UsuarioResponseDTO.fromEntity(usuario);
    }

    @Override
    public void eliminar(UUID id) {
        if (!repo.existsById(id)) {
            throw new ResponseStatusException(              // 404 Not Found
                HttpStatus.NOT_FOUND, "Usuario no encontrado: " + id);
        }
        repo.deleteById(id);
    }
}

El orden importa

1) mapear DTO→entidad · 2) cifrar la contraseña · 3) validar y comprobar correo único · 4) guardar y devolver DTO. Si algo falla en el paso 3, nunca se llega a guardar.

Referencia

Códigos de estado HTTP

Toda respuesta HTTP trae un número de 3 dígitos. El primer dígito marca la familia: así sabes de un vistazo si algo salió bien, si es culpa del cliente o del servidor.

2xx · Éxito

La petición se procesó bien.
200 OK 201 Created 204 No Content

3xx · Redirección

El recurso está en otra parte o no cambió.
301 Moved 302 Found 304 Not Modified

4xx · Error del cliente

La petición está mal: falta algo, formato inválido, no existe.
400 401 403 404 409

5xx · Error del servidor

Falló el servidor, no el cliente.
500 502 503

Confusiones típicas

  • 401 vs 403: 401 = “no sé quién eres” (falta login); 403 = “sé quién eres, pero no puedes”.
  • 400 vs 422: 400 = la petición está mal formada; 422 = está bien formada pero los datos no pasan las reglas. Muchos equipos usan 400 para todo por simplicidad (como aquí).
  • 200 vs 201 vs 204: hay cuerpo y ya existía → 200; se creó algo → 201; éxito sin cuerpo → 204.

La regla mental

4xx = culpa del cliente → “arregla tu petición”. 5xx = culpa del servidor → “el backend falló”. Con ResponseStatusException tú eliges el 4xx correcto; el 500 aparece solo cuando algo se te escapa sin controlar.

ResponseStatusException · el status manda 
// 400 · dato inválido (en el validador)
throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "...");

// 404 · no existe (en el servicio)
throw new ResponseStatusException(HttpStatus.NOT_FOUND, "...");

// 409 · conflicto: correo duplicado (en el servicio)
throw new ResponseStatusException(HttpStatus.CONFLICT, "...");

Tip: el status del éxito también se elige

Para el 201 al crear y el 204 al borrar, el controlador (siguiente tutorial) usa @ResponseStatus(HttpStatus.CREATED) o devuelve un ResponseEntity. ResponseStatusException es solo para los errores.

Glosario rápido

Interfaz (I…)

El contrato: qué métodos existen, sin el cómo. IUsuarioValidador, IUsuarioServicio.

…Impl

La implementación del contrato. Aquí vive la lógica.

@Component / @Service

Registran la clase como bean para que Spring la inyecte. @Service = negocio.

Inyección por constructor

Dependencias como private final recibidas en el constructor. Sin @Autowired en campo.

DTO

Objeto de transporte. Request (entra) y Response (sale). Nunca la entidad.

toEntity / fromEntity

Conversión POO sin mapper: DTO→entidad y entidad→DTO, a mano.

ResponseStatusException

Excepción de Spring Web. Lanzas el status HTTP directo: 400 inválido, 404 no existe, 409 duplicado.

HttpStatus

Enum con los códigos: BAD_REQUEST, NOT_FOUND, CONFLICT, CREATED…

PasswordEncoder

Cifra la contraseña con BCrypt → un hash de 60 caracteres. La entidad guarda el hash, no la clave.

Regla de oro del día

Cada capa con una responsabilidad: el validador valida, el DTO transporta, el servicio orquesta. Se conectan con interfaces e inyección por constructor, y la contraseña jamás sale por la API.
